# Eigenstate 2 Geliştirme Görevleri

Bu sayfa, Eigenstate 2 içindeki GitHub ve geliştirme odaklı görevler için topluluk rehberidir.

Geliştirme kategorisi; QoreChain organizasyonunu takip etme, repoları yıldızlama, fork alma, anlamlı issue açma, pull request gönderme ve merge edilmiş PR kanıtı sunma gibi görevleri kapsar.

## Kapsam

Gözlemlenen görev yapısına göre bu kategoride 6 görev bulunur. Bu görevler az sayıda görünse de, özellikle issue ve PR görevleri topluluk katkısı açısından en görünür alanlardan biridir.

## Görev Türleri

| Görev | Kanıt İçin En Uygun Link |
|---|---|
| QoreChain organizasyonunu takip et | GitHub profiliniz veya QoreChain organizasyon sayfası |
| Ana repoyu yıldızla | İlgili GitHub repo linki |
| 3+ QoreChain reposunu yıldızla | Profil, repo listesi veya ilgili repo linkleri |
| Docs reposunu forkla | Fork edilmiş repo linkiniz |
| Meaningful issue aç | Açtığınız issue linki |
| Pull request gönder / merge edilmiş PR | PR linki |

## Meaningful Issue Nedir?

Anlamlı issue, yalnızca görev tamamlamak için açılmış gibi görünmemelidir. Gerçek bir eksikliği, belirsizliği, hata ihtimalini veya iyileştirme önerisini açıklamalıdır.

İyi bir issue genellikle şunları içerir:

- Net başlık
- Kısa problem açıklaması
- Hangi dosya, bölüm veya davranışla ilgili olduğu
- Neden önemli olduğu
- Uygulanabilir öneri

Zayıf issue örnekleri:

- "Please update docs"
- "Good project"
- "Need more info"

Daha güçlü issue örneği:

```text
Improve documentation for RPC endpoints and network configuration
```

## Pull Request İçin Notlar

Pull request, kod değişikliği olmak zorunda değildir. Dokümantasyon düzeltmesi, örnek yapılandırma dosyası, README iyileştirmesi veya bağlantı düzeltmesi de değerli olabilir.

PR açıklamasında şunları belirtmek yararlı olur:

- Ne değişti?
- Neden değişti?
- Kod, yapılandırma veya runtime davranışı değişti mi?
- Dokümantasyon dışı bir risk var mı?

Örnek açıklama:

```text
This is a documentation-only change. It improves clarity for new operators and does not modify code, configuration, or runtime behavior.
```

## Merge Edilmiş PR Kanıtı

Bir PR yalnızca GitHub üzerinde merge edildiyse "merged PR" olarak sunulmalıdır. Açık PR için "submitted pull request" veya "open pull request" ifadesi daha doğrudur.

## Güvenlik Notları

- Özel anahtar, token, IP, sunucu erişim bilgisi veya gizli yapılandırma paylaşmayın.
- Config örneği verirken gerçek credential kullanmayın.
- Başkasının açtığı issue veya PR'ı kendi katkınız gibi göstermeyin.
- Resmi repo üzerinde gereksiz issue açarak spam görüntüsü oluşturmayın.
