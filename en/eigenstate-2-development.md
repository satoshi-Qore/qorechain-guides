# Eigenstate 2 Development Tasks

This guide covers the Development category in QoreChain Eigenstate 2.

Observed category size: 6 tasks.

Development tasks usually involve GitHub actions such as following the QoreChain organization, starring repositories, opening issues, submitting pull requests, and providing meaningful documentation or code-related contributions.

## Purpose

The goal of Development tasks is to encourage useful open-source participation. A good development submission should be public, relevant, and easy to review.

## Common Task Types

| Task Type | Typical Proof | Notes |
|---|---|---|
| Follow organization | GitHub profile or organization page | A screenshot may help if the action is not public |
| Star repositories | Repository links or profile stars page | Use official QoreChain repositories when required |
| Fork repository | Fork URL | Make sure the fork is public if used as proof |
| Open meaningful issue | Issue link | The issue should describe a real problem or improvement |
| Submit pull request | PR link | Do not call it merged before merge happens |
| Merged PR | Merged PR link | Use only after GitHub shows the PR as merged |

## What Makes an Issue Meaningful?

A meaningful issue should include:

- a clear title;
- a specific problem or improvement;
- context or reproduction notes if relevant;
- a practical suggestion;
- neutral and respectful wording.

## What Makes a PR Useful?

For documentation PRs, useful contributions include:

- fixing unclear setup steps;
- adding missing example files;
- improving README navigation;
- correcting outdated wording;
- adding safe configuration examples;
- improving troubleshooting notes.

## Description Examples

```text
I opened a meaningful GitHub issue describing a documentation improvement and submitted the issue link for review.
```

```text
I submitted a documentation pull request and provided the public PR link for manual review.
```

```text
The pull request has been merged, and I submitted the merged PR link as proof.
```

## Risk Notes

- Do not open low-effort or duplicate issues.
- Do not submit noisy PRs just for task completion.
- Do not claim official status.
- Do not describe an open PR as merged.
